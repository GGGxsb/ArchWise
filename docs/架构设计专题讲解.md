# ArchWise 架构设计专题讲解

## 从一个问题开始：鲜花门店该用什么架构？

有人问 DeepSeek："鲜花门店管理系统，录入鲜花品类库存定价，顾客线上下单配送，破损登记损耗台账，月末汇总营收与损耗。用什么架构？"

DeepSeek 答案是微服务。这个答案"技术上正确"——微服务确实可以做任何事。但把一个单体就能完美承载的系统拆成 6 个独立部署的服务，引入了多少成本？

这个问题的本质是：**LLM 倾向于推荐"看起来先进"的架构，而忽视了维护成本。** ArchWise 的设计，很大程度上是为了解决这个问题。

---

## 一、核心设计理念：架构倾向而非架构规则

### 规则引擎为什么不行

最初的设计有一个规则引擎，逻辑是：

```
if concurrency < 0.35 and realtime < 0.35:
    偏好单体/分层，排斥微服务
```

问题在于：`concurrency` 和 `realtime` 也是 LLM 提取的。如果 LLM 把鲜花门店的并发标为 0.6（认为"顾客同时浏览算并发"），规则引擎就被绕过了。它没有引入**独立的判断能力**。

### Prompt 为什么更合适

现在的做法是把约束写成自然语言注入 Prompt：

> 简单架构能实现的需求就不应考虑复杂架构。分层、MVC、单体等方案能胜任时，不应为了「架构看起来先进」而引入微服务、事件驱动、CQRS 等重型方案。

LLM 在语义空间中判断"这个需求是否简单"，而不是比对数值阈值。效果是：鲜花门店被正确识别为"简单系统"，但万人即时通讯系统仍然能拿到事件驱动推荐。

**关键区别**：规则引擎把"简单"编码为 `concurrency < 0.35`，Prompt 让 LLM 用常识判断"鲜花门店"是否属于简单系统。后者的容错空间更大。

## 二、三 Agent 的设计哲学

### 不是"三个工人"，而是"三道工序"

很多多 Agent 系统的设计是平级的：多个 Agent 各自给出意见，然后投票或协商。这种模式适合需要多视角的问题（如辩论、创意生成），但架构推荐是一个**串行决策链**——每个阶段的输入严格依赖前一阶段的输出。

所以 ArchWise 的 Agent 是管道式、非平级的：

```
自然语言需求
    │
    ▼ RequirementParserAgent      ← 只负责"理解"
    │  LLM 提取 + 本地校验
    ▼ ExtractedFeatures
    │
    ▼ ArchitectureMatcherAgent    ← 只负责"匹配"
    │  LLM 评分 + 后置校验
    ▼ 候选排序列表
    │
    ▼ EvaluationGeneratorAgent    ← 只负责"表达"
    │  LLM 生成 + 模板兜底
    ▼ Markdown 报告
```

每个 Agent 解决一个独立子问题，互不越界。

### 为什么 LLM 调用和后置校验分开

ArchitectureMatcherAgent 的做法是：

1. **LLM 先独立评分**（基于语义理解）
2. **本地再做一致性校验**（基于确定性规则）

为什么不反过来？如果先跑本地评分再告诉 LLM"本地认为 A 比 B 好"，LLM 会被前置偏差引导，丧失独立判断能力。后置校验保留了 LLM 的推理空间，只在出现显著矛盾时才标记——这更接近"同行评审"而非"上级指令"。

### 什么情况下本地校验会介入

```python
# 场景 1：首选项分歧
LLM 首选: 微服务(92分)
本地首选: 单体分层(86分)
→ LLM 首选的微服务在本地评分中只有 74 分
→ 标记：降置信度，追加风险提示

# 场景 2：过度设计
需求特征: concurrency=0.15, realtime=0.2, 业务能力 7 项
LLM 将微服务列为"核心推荐"
→ 标记：过度设计预警
→ 追加扣分原因："需求规模与架构复杂度不匹配"

# 场景 3：前三全是重型
简单系统 + 前三候选: 微服务、事件驱动、CQRS
→ 追加提示："前三名均不包含简单架构，建议考虑是否过度设计"
```

## 三、6 种架构风格的分类逻辑

### 为什么是 6 种

原来的 12 种存在三个结构性问题：

**1. 概念重叠**。单体架构描述的是"如何部署"，分层架构描述的是"如何组织代码"，MVC 描述的是"表现层如何与业务交互"。它们是正交维度，却被当作互斥选项。实际上，一个系统可以同时是单体 + 分层 + MVC——这就是现在的 `monolithic_layered`。

**2. 决策疲劳**。12 个选项对目标用户（学生、小团队做架构决策）没有帮助。他们不需要在六边形架构和整洁架构之间纠结——两个都是以领域为中心的隔离模式，选一个就够了。

**3. 死风格**。SOA（遗留企业集成）、黑板（极窄的 AI 诊断场景）、Serverless（是部署选择不是软件架构）在 95% 的需求中不适用。

### 合并逻辑

| 原风格 | 去向 | 理由 |
|--------|------|------|
| monolithic + layered + MVC | → monolithic_layered | 部署方式 + 代码组织 + 表现层模式，同一方案的不同侧面 |
| clean architecture | → 合并到 hexagonal | 同为领域中心架构，hexagonal 更普及 |
| SOA | → 删除 | 遗留企业集成模式，绿场项目不适用 |
| blackboard | → 删除 | 极窄场景（AI 诊断系统） |
| serverless | → 删除 | 是部署/基础设施选择，非软件体系结构模式 |

### 六种的职责划分

```
简单 ──────────────────────────────────────► 复杂
  │                                            │
  ▼                                            ▼
monolithic_layered                      microservices
  (单体分层)                              (微服务)
                                         event_driven
                                           (事件驱动)
                                         cqrs
                                           (读写分离)
                                         
数据流处理 ─────────────────────────► pipe_filter
                                      (管道-过滤器)

领域复杂 / 长期演进 ─────────────────► hexagonal
                                      (六边形架构)
```

## 四、LLM 集成的工程设计

### 调用次数设计

一次完整推荐 = 3 次 LLM 调用（+ 拓扑时 4 次）。这是有意控制的：

- 把"需求理解 + 架构匹配 + 报告生成"拆成三次独立调用，而非一次巨型 prompt。好处是每次调用的 prompt 短、目标明确、出错后可以针对性重试。
- 不把本地逻辑也交给 LLM（如 JSON 解析、Schema 校验、一致性检查、模板格式化），减少 Token 消耗和幻觉风险。

### Prompt 结构原则

每次 Prompt 遵循相同结构：角色设定 → 输出格式约束 → 语义约束 → 输入数据。

```
你是 [角色]。
只返回 JSON，不要 Markdown，不要解释。

约束：
1. [格式约束]
2. [语义约束]
3. ...

[输入数据]
```

这种结构让 LLM 的行为可预测：它知道自己是什么角色、应该输出什么格式、遵循什么规则、基于什么数据。

### 输出可靠性

| 风险 | 应对 |
|------|------|
| LLM 返回 Markdown 包裹的 JSON | `_extract_json()` 用正则剥离 ``` 代码块 |
| JSON 字段缺失 | Pydantic 校验拒绝不完整输出 |
| JSON 包含编造的 style_id | `_sanitize_architecture_recommendation()` 过滤 style_id 不在知识库中的条目 |
| 需求解析返回空特征 | 切换到严格模式重试；仍失败则使用正则+关键词兜底 |
| 报告缺少章节 | `_check_completeness()` 检测缺失并本地模板补全 |

## 五、设计取舍与反思

### 做的事

- ✅ 三个 Agent 分工明确，串行管道不越界
- ✅ Prompt 编码架构倾向而非硬编码规则
- ✅ LLM 做主判断，本地做校验和兜底
- ✅ 6 种架构风格覆盖主流场景且无重叠
- ✅ 输出可追溯（需求特征 → 评分证据 → 最终推荐）

### 不做的事

- ❌ 不用 Neo4j 存储架构知识（6 种风格的内存结构已足够）
- ❌ 不用规则引擎替代 LLM 判断（阈值脆弱且未引入独立能力）
- ❌ 不让 Agent 平级投票（架构推荐是串行决策，不是多视角讨论）
- ❌ 不在 Prompt 里写具体场景映射（"鲜花门店用 MVC"是脆弱的，原则性约束更通用）

### 如果要做下一步

- **LLM 缓存**：相同需求重复请求时返回缓存结果
- **人工反馈回路**：用户标记"这个推荐不对"后，反馈存入案例库
- **案例相似度检索**：基于历史案例的 embedding 相似度做推荐增强
- **多 LLM 交叉验证**：用 Qwen 做一致性检查，当两个 LLM 的推荐差异过大时发出警告
