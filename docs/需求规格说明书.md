# ArchWise 需求规格说明书

## 1. 项目目标

ArchWise 面向“软件体系结构风格智能助手”课程作业，接收用户自然语言软件需求，自动完成语义特征抽取、候选架构推荐、多维度对比、最终决策报告生成，并支持知识库扩展与案例学习。

## 2. 用户与使用场景

- 学生或架构学习者：输入需求，理解不同架构风格的适用场景。
- 课程验收人员：通过 Web 页面或 Web API 验证系统是否具备 Agent、LLM、知识库和可视化能力。
- 架构知识维护者：补充新架构风格、新案例和适用规则。

## 3. 功能性需求

| 编号 | 需求 | 说明 |
| --- | --- | --- |
| FR-01 | 需求理解 | 识别领域、关键词、并发、实时性、可靠性、扩展性、数据密集度、AI 推理等特征 |
| FR-02 | 架构推荐 | 至少输出 3 种候选体系结构风格 |
| FR-03 | 决策支持 | 输出评分、推荐理由、风险、优缺点和最终推荐 |
| FR-04 | 知识库 | 内置不少于 10 种架构风格属性、适用场景、拓扑图和规则 |
| FR-05 | 混合推理 | 基于规则引擎评分，并在可用时调用 LLM 生成报告 |
| FR-06 | 可视化 | 展示对比矩阵、Agent 协作追踪和 Mermaid 架构拓扑图 |
| FR-07 | 知识进化 | 支持通过 API 新增架构风格和案例 |
| FR-08 | Web API | 提供推荐、知识库查询、知识图谱查询和案例写入接口 |

## 4. AI 系统特有需求

- 可解释性：每个候选架构必须包含命中规则和风险原因。
- 可追溯性：输出 Agent 协作 trace，说明从需求解析到报告生成的步骤。
- 鲁棒性：LLM 不可用或无 API Key 时，系统必须回退到本地规则和模板报告。
- 可控性：LLM 参与需求增强、候选复核和报告生成，但最终候选评分仍由规则引擎约束，降低幻觉风险。
- 知识进化：新增案例和架构风格后应立即影响后续查询或可用于扩展训练数据。

## 5. 非功能需求

- 响应时间：普通文本推荐在本地规则模式下应在 1 秒内完成。
- 可扩展性：知识库采用结构化模型，可替换为 Neo4j 等图数据库。
- 可维护性：Agent、知识库、LLM 适配器、API 分层实现。
- 可演示性：无需 API Key 也可完整运行；有 Key 时可展示真实 LLM 集成。
