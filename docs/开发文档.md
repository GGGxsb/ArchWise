# ArchWise 开发文档

## 第一部分：需求规格说明书

### 1. 引言

#### 1.1 项目背景

在软件系统设计早期，开发者需要根据业务规模、并发压力、实时性、可靠性、扩展性等约束选择合适的软件体系结构风格。传统方式依赖人工经验，容易出现判断主观、依据不清晰、候选方案对比不足的问题。

ArchWise 将"自然语言需求"转化为"结构化架构决策"，通过大语言模型和多 Agent 协作，提供可解释、可追溯、可视化的架构推荐结果。

#### 1.2 适用场景

- 软件体系结构课程教学与作业辅助
- 软件系统早期架构选型
- 需求到架构的映射演示与验证
- 多架构方案对比分析
- 架构设计原型快速验证

#### 1.3 术语定义

| 术语 | 说明 |
|------|------|
| 架构风格 | 软件体系结构的组织模式，如单体分层、微服务、事件驱动 |
| Agent | 具有独立职责的智能处理单元，在此系统中指需求解析、架构匹配、评估生成三个 Agent |
| LLM | Large Language Model，本系统使用 DeepSeek API |
| 拓扑图 | 以 Mermaid 图形式展示的架构组件和连接关系 |
| 决策溯源 | 展示从需求特征到最终推荐的全链路证据和推理过程 |

### 2. 功能性需求

#### 2.1 需求理解 (F-REQ-01)

**描述**：系统接收自然语言描述的软件需求，提取结构化特征。

**输入**：自然语言文本（最短 5 个字符，最长 8000 个字符）

**输出**：
- `domain`：业务领域分类
- `keywords`：技术及业务特征关键词
- `business_capabilities`：具体业务能力列表（如"录入鲜花品类""顾客下单""配送签收"）
- `quality_attributes`：6 个质量属性的 0~1 量化值（concurrency、realtime、reliability、scalability、data_intensity、ai_reasoning）
- `data_flow`：数据流类型（event_stream / pipeline / transactional / request_response）
- `constraints`：部署约束、高可用要求、扩展性诉求
- `topology_expectations`：期望的组件和关系
- `ambiguity_notes`：模糊性和一致性警告

**验收标准**：
- 输入为空或纯关键词时拒绝并提示
- LLM 不可用时使用正则+关键词模板兜底
- 内部一致性校验通过（领域与关键词匹配、质量属性与业务能力规模匹配）

#### 2.2 架构推荐 (F-REQ-02)

**描述**：基于结构化需求特征和 6 种候选架构风格，推荐最优架构并生成多维度对比。

**输出**：
- 候选架构列表（按评分降序），每个候选包含：综合评分（0~100）、推荐定位（核心推荐/备选方案/专项补充/不推荐）、置信度、匹配理由、风险、扣分原因、6 维度质量评分
- 架构组合建议（是否需要组合架构、主架构和辅助架构各负责哪些部分）
- 过度设计预警

**验收标准**：
- 至少返回 3 个候选架构
- 简单系统不推荐微服务/事件驱动/CQRS
- 高并发或高实时系统优先推荐相应的重型架构
- 评分合理拉开差距

#### 2.3 评估报告 (F-REQ-03)

**描述**：生成可读的 Markdown 评估报告，包含推荐理由、多维度对比、风险提示和落地建议。

**验收标准**：
- 包含推荐理由、风险提示、落地建议三个必要章节
- LLM 不可用时使用本地模板生成兜底报告
- 支持流式输出（SSE），逐 token 推送报告内容

#### 2.4 拓扑图生成 (F-REQ-04)

**描述**：根据推荐架构风格和需求特征，生成 Mermaid 架构拓扑图，包含业务组件、数据存储和连接关系。组件正确映射到架构风格的层结构，数据存储标注为 data_store 类型并归属到业务层。

#### 2.5 知识库管理 (F-REQ-05)

**描述**：支持架构风格和典型案例的查询与扩展。

| 类别 | 要求 |
|------|------|
| 性能 | 首次推荐响应 ≤ 15 秒（含 3~4 次 LLM 调用）；流式报告首 token ≤ 5 秒 |

| 可维护性 | 架构风格声明式定义；Agent 职责独立，修改不互相影响 |
| 可追溯性 | 每次推荐保留完整决策追踪链 |

### 4. AI 系统特有需求

#### 4.1 Prompt 工程要求 (AI-PROMPT-01)

**需求解析 Prompt**：禁止返回空业务能力列表；数据存储组件必须标注 type=data_store / layer=data；业务能力名称必须具体（禁止"业务处理""数据处理"等泛化词）；支持两轮提取（首次宽松 + 严格重试）。

**架构匹配 Prompt**：简单架构能实现的需求不考虑复杂架构；每个方案必须评估技术债和维护成本；仅在高并发/强实时/多团队等硬需求明确存在时推荐重型架构；返回严格 JSON，禁止 Markdown 包裹。

#### 4.2 LLM 输出可靠性保障 (AI-RELI-01)

| 策略 | 说明 |
|------|------|
| JSON Schema 约束 | 使用 `response_format: json_object` 确保结构化输出 |
| 两次提取 | 需求解析失败时自动切换到严格模式重试 |
| Pydantic 校验 | 所有 LLM 输出经 Pydantic Schema 验证 |
| 一致性检查 | RequirementParserAgent 检查领域-关键词-能力链的内部自洽性 |
| 兜底机制 | LLM 完全不可用时切换到本地模板和规则 |

#### 4.3 Agent 协作要求 (AI-AGENT-01)

三个 Agent 以链式管道协作，前一阶段的输出作为下一阶段的输入：

1. **RequirementParserAgent**：输入校验 → LLM 提取 → 一致性检查 → 兜底
2. **ArchitectureMatcherAgent**：LLM 评分 → 本地后置校验 → 过度设计检测
3. **EvaluationGeneratorAgent**：LLM 生成 → 章节完整性 → 兜底模板

Agent 之间不直接通信，通过 Pipeline 上下文传递数据。

#### 4.4 架构倾向约束 (AI-TENDENCY-01)

系统通过 Prompt 编码架构设计倾向而非硬编码规则：简单架构能实现的需求不应考虑复杂架构；单体分层的默认优先级高于微服务；每个推荐必须评估复杂度代价和团队匹配度；避免"为了架构看起来先进"引入技术债。

### 7. 三 Agent 协作机制

#### 7.1 设计原则

**问题分解而非职责堆叠**。每个 Agent 解决架构推荐流程中的一个独立子问题，形成严格的串行管道，前一阶段的输出是后一阶段的输入。Agent 之间不直接通信，通过 Pipeline 上下文传递数据。

#### 7.2 RequirementParserAgent（需求解析 Agent）

```
parse(requirement)
  ├── 输入校验：非空、长度、自然语言判定
  ├── LLM 提取 #1：DeepSeek extract_features()
  │     └── 两轮策略：宽松 → 严格重试
  ├── 本地一致性检查
  │     ├── 领域 vs 关键词自洽性
  │     ├── 高并发 + 小系统不匹配检测
  │     ├── AI标记 + 无AI关键词检测
  │     └── data_flow vs 业务能力一致性
  └── 兜底：正则 + 关键词模板
        └── 6 组正则模式覆盖领域、质量属性、数据流类型
```

**为什么需要本地一致性检查**：LLM 可能返回内部矛盾的结果（如 domain="电商"但 keywords 全是医疗术语，或 concurrency=0.9 但 business_capabilities 只有 3 项）。本地校验不替代 LLM 判断，但在矛盾时记录 ambiguity_notes，为下游 Agent 和用户提供透明度。

#### 7.3 ArchitectureMatcherAgent（架构匹配 Agent）

**双重评估模式**：LLM 做主评分，本地做后置校验。

```
LLM 主评分：
  recommend_architectures(requirement, features, styles)
    → DeepSeek 调用 #2
    → 返回候选排序 + 评分 + 组合建议

本地后置校验：
  validate_llm_results(features, llm_candidates, styles)
    ├── 本地 match() 跑一遍作为基线
    ├── 首选项分歧检测 → 降置信度、标记风险
    ├── 简单系统推重型架构 → 过度设计预警
    └── 前三全是重型架构 → 追加提示
```

**为什么是后置校验而非前置**：前置校验（先跑本地评分再喂给 LLM）会让 LLM 被本地偏差引导，丧失独立判断能力。后置校验保留 LLM 的语义理解优势，只在出现显著矛盾时标记——这更接近"同行评审"而非"上级指令"。

**本地校验介入场景**：

| 场景 | 检测条件 | 动作 |
|------|----------|------|
| 首选项分歧 | LLM 首选 ≠ 本地首选，且 LLM 首选在本地位次较低 | 降置信度，追加风险提示 |
| 过度设计 | 简单系统（低并发+低实时+小规模），LLM 将重型架构列为核心推荐 | 标记过度设计预警，追加扣分原因 |
| 缺少简单选项 | 前三候选全是重型架构（微服务/事件驱动/CQRS） | 追加提示"建议考虑简单架构" |

**本地评分机制**：基于质量属性加权求和（`ATTRIBUTE_MAP`）+ 数据流类型加分（`FLOW_BONUS`）+ 上下文适配调整（`_context_fit_adjustment`）。

#### 7.4 EvaluationGeneratorAgent（评估生成 Agent）

```
generate(requirement, features, candidates)
  ├── LLM 调用 #3: generate_report()
  ├── 章节完整性检查
  │     ├── 推荐理由 → 缺失时用候选评分补充
  │     ├── 风险提示 → 缺失时用候选风险补充
  │     └── 落地建议 → 缺失时用模板补充
  └── 完全不可用时 → 全量本地模板
```

报告生成是 LLM 的强项（自然语言组织），但 LLM 可能遗漏章节。本地模板作为结构性兜底，只补充缺失部分，不替代已有内容。

### 8. LLM 集成方案

#### 8.1 调用点设计

一次完整的推荐链路进行 **3 次 LLM 调用**（拓扑生成额外 1 次）：

| # | 方法 | 作用 | 输入 | 输出 |
|---|------|------|------|------|
| 1 | `extract_features` | 需求 → 结构化特征 | 自然语言文本 | ExtractedFeatures JSON |
| 2 | `recommend_architectures` | 特征 → 候选排序 | 特征 + 6 种风格知识库 | 候选列表 + 组合建议 JSON |
| 3 | `generate_report` | 候选 → 评估报告 | 需求 + 特征 + 候选列表 | Markdown 报告片段 |
| 4 | `extract_style_instance` | Schema → 组件填充 | 需求 + 特征 + StyleSchema | StyleInstance JSON |

把"需求理解 + 架构匹配 + 报告生成"拆成三次独立调用而非一次巨型 prompt，好处是每次调用的 prompt 短、目标明确、出错后可以针对性重试。

#### 8.2 Prompt 结构原则

每次 Prompt 遵循相同结构：**角色设定 → 输出格式约束 → 语义约束 → 输入数据**。让 LLM 的行为可预测。

**架构匹配 Prompt 中的架构倾向约束**（核心）：

> 简单架构能实现的需求就不应考虑复杂架构。分层、MVC、单体等方案能胜任时，不应为了「架构看起来先进」而引入微服务、事件驱动、CQRS 等重型方案。优先选择能满足需求的最简架构。

> 每个推荐方案必须深度评估技术债和长期维护成本：引入的复杂度代价、团队规模匹配度、数据一致性代价、部署与调试成本。

> 只有在高并发、强实时、多团队独立交付、或最终一致性等硬需求明确存在时，才可以推荐微服务、事件驱动或 CQRS。

#### 8.3 LLM 客户端设计

```python
class LLMClient:
    """OpenAI-compatible adapter. 支持 DeepSeek、Qwen 等兼容网关。"""
    
    # 核心方法
    async def extract_features(requirement) -> ExtructuredFeatures | None
    async def recommend_architectures(requirement, features, styles, top_k) -> ...
    async def generate_report(requirement, features, candidates) -> str | None
    async def extract_style_instance(requirement, features, schema) -> StyleInstance | None
    
    # 配置
    api_key: str     # 从 LLM_API_KEY 或 DEEPSEEK_API_KEY 环境变量读取
    base_url: str    # 默认 https://api.deepseek.com
    model: str       # 默认 deepseek-v4-flash
    timeout: float   # 默认 12 秒
```

#### 8.4 输出可靠性策略

| 风险 | 应对 |
|------|------|
| LLM 返回 Markdown 包裹的 JSON | `_extract_json()` 用正则剥离 ``` 代码块 |
| JSON 字段缺失 | Pydantic 校验拒绝不完整输出 |
| JSON 包含编造的 style_id | `_sanitize_architecture_recommendation()` 过滤不在知识库中的条目 |
| 需求解析返回空特征 | 切换到严格模式重试；仍失败则使用正则+关键词兜底 |
| 报告缺少章节 | `_check_completeness()` 检测缺失并本地模板补全 |
| thinking 拖慢响应 | 所有调用设置 `"thinking": {"type": "disabled"}` |

### 9. 核心数据流

#### 9.1 同步推荐

```
POST /api/recommend { requirement, top_k }
  │
  ├── RequirementParserAgent.parse(requirement)
  │     └── DeepSeek #1 → ExtractedFeatures
  │
  ├── ArchitectureMatcherAgent (via orchestrator)
  │     ├── DeepSeek #2 → candidates + composition
  │     └── validate_llm_results() → consistency notes
  │
  ├── EvaluationGeneratorAgent.generate()
  │     └── DeepSeek #3 → Markdown report
  │
  └── _build_topologies()
        └── DeepSeek #4 → Mermaid diagram
```

#### 9.2 流式推荐

```
POST /api/recommend/stream { requirement, top_k }
  │
  ├── SSE: features → 需求特征
  ├── SSE: recommendation → 候选排序 + 决策溯源
  ├── SSE: report_delta × N → 逐段推送报告
  └── SSE: done → 完成
```

### 10. 知识表示

#### 10.1 架构风格知识库

6 种架构风格以 Python 字典声明式定义，每个风格包含 id、name、category、description、suitable_for、quality_scores（6 维 0~1 量化值）、strengths、weaknesses、topology（Mermaid 模板）、rules（prefer/avoid 关键词）、schema_id（指向拓扑 Schema）。

#### 10.2 拓扑 Schema

`style_schemas.py` 为每种架构风格定义：
- **LayerSpec**：层的 ID、名称、描述、是否必需、最小组件数、是否单例
- **LayerConnectionRule**：层间连接规则（源层、目标层、连接类型、是否允许跨层）
- **Topology Template**：Mermaid 模板骨架，含 `%%LAYER%%` 和 `%%EDGES%%` 占位符
- **Prompt Hints**：注入 LLM 提取 prompt 的额外约束

LLM 将架构 Schema 与用户需求结合，生成属于该风格的实例化组件和连接。

### 11. 关键设计决策

#### 11.1 为什么是 6 种架构风格

原来的 12 种存在概念重叠（单体与分层不是互斥选择）、决策疲劳（12 个选项对学生没有帮助）、死风格（SOA、黑板、Serverless 在 95% 场景不适用）。合并策略：monolithic + layered + MVC → monolithic_layered；clean → hexagonal；SOA、blackboard、serverless → 删除。

#### 11.2 为什么不用 Neo4j 知识图谱

架构知识已被 StyleSchema 结构化，LLM 直接通过 prompt 消费；没有持续增长的历史案例库需要图遍历；6 种风格的关系图谱用内存结构即可表达。删除约 1000 行 Neo4j 代码后，系统复杂度明显降低。

#### 11.3 为什么不用本地规则引擎

规则引擎依赖 LLM 提取的 quality_attributes 数值，并未引入独立的判断能力——如果 LLM 把鲜花门店的 concurrency 标为 0.6，规则引擎的 0.35 阈值就被绕过了。Prompt 中的自然语言约束让 LLM 在语义空间中做判断，更灵活且不依赖脆弱的数值阈值。

#### 11.4 架构倾向而非架构规则

系统不通过硬编码判断"什么场景用什么架构"。Prompt 编码通用原则（"简单架构能做就不用复杂架构"），让 LLM 根据语义理解自行判断具体场景。这比"concurrency < 0.35 → 偏好单体"更灵活——LLM 能理解"鲜花门店"的语义，而规则引擎只能比对数值。

### 12. 项目目录结构

```text
ArchWise/
  app/
    agents/
      requirement_parser.py    # Agent 1: 需求解析 + 一致性校验 + 兜底
      architecture_matcher.py  # Agent 2: LLM 评分 + 本地后置校验
      evaluation_generator.py  # Agent 3: 报告生成 + 模板兜底
    knowledge/
      styles.py                # 6 种架构风格定义
      style_schemas.py         # 6 种拓扑 Schema (层/连接/模板)
      repository.py            # 知识库 CRUD
    models/
      schemas.py               # Pydantic 数据模型
    services/
      llm_client.py            # OpenAI-compatible LLM 客户端
      hybrid_orchestrator.py   # 三 Agent 链式编排器
      recommendation_service.py # 推荐服务入口
      composition_recommender.py # 组合架构推荐
      report_formatter.py      # Markdown 报告格式化
      style_topology_renderer.py # 拓扑图 Mermaid 渲染
      knowledge_graph.py       # 内存图构建器 (前端可视化)
    static/                    # CSS + JS
    templates/                 # Jinja2 页面模板
    main.py                    # FastAPI 入口
  docs/
    开发文档.md                 # 需求规格 + 架构设计（本文档）
    架构设计专题讲解.md         # 设计哲学与关键决策
  tests/
  requirements.txt
  .env.example
  README.md
```
